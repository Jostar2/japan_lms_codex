# Codex to Claude: AI Affordance / UX Handoff

## 목적

이 문서는 Claude가 다음 라운드에서 `xAI_LMS_Prototype.html`과 `xAI_LMS_Design_Spec.html`을 고도화할 때 반드시 반영해야 할 Codex 측 UX 판단이다. 스타일을 크게 갈아엎는 목적이 아니라, 현재 Claude가 잡아둔 화면 구조를 무너뜨리지 않으면서 "어떤 항목이 AI와 상호작용 가능한지"를 더 명확하고 깊이 있게 만드는 것이 목적이다.

## 핵심 문제 인식

현재 화면은 AI가 있는 항목을 배지, 문구, 작은 마커로 표시하는 경향이 강하다. 이것만으로는 사용자가 다음을 한눈에 이해하기 어렵다.

- AI가 무엇을 감지했는가
- AI가 어떤 판단을 내렸는가
- AI가 무엇을 추천/생성/검증했는가
- 사용자가 지금 무엇을 할 수 있는가
- 사용자가 AI 판단을 어떻게 거부하거나 수정할 수 있는가

따라서 단순히 `AI` 배지를 많이 붙이는 방식은 피해야 한다. AI affordance는 색상이나 텍스트만이 아니라 형태, 위치, 행동, 상태 변화로 구분되어야 한다.

## Codex가 제안하는 5개 AI Affordance Grammar

### 1. Signal Marker = AI 감지

AI가 위험, 병목, 이상 패턴, 학습 상태 변화를 감지한 항목이다.

UI 원칙:
- 작은 배지만 붙이지 말고, 항목의 좌측 edge, 상태 점, 얇은 rail, pulse dot 등으로 "감지됨"을 표시한다.
- 이 요소는 바로 실행 CTA가 아니라 "왜 감지됐는지 열어볼 수 있음"을 암시해야 한다.
- 예: 부담 리스크, 이해도 하락, 제출 지연, 불확실 채점 후보.

권장 문법:
- `감지 근거 보기`
- `AI 감지: 부담 리스크`
- confidence 또는 threshold를 함께 노출한다.

### 2. Decision Rail = AI 판단

AI가 여러 항목 중 우선순위를 정했거나, 오늘 무엇을 먼저 해야 하는지 판단한 항목이다.

UI 원칙:
- 카드 내부에 단순 배지를 붙이지 말고, 판단 카드 옆에 세로 decision rail 또는 priority strip을 둔다.
- rail에는 우선순위, 판단 신뢰도, 근거 수, uncertainty를 압축 표시한다.
- 클릭 시 우측 AI 패널의 특정 xAI context가 열려야 한다.

권장 문법:
- `왜 1순위인가`
- `판단 근거`
- `대안 보기`
- `교수자/학생이 최종 결정`

### 3. Action Stack = AI 추천

AI가 다음 행동을 추천하는 영역이다.

UI 원칙:
- 버튼 하나로 끝내지 말고, "추천 행동 1-2개 + 보류/거부/수정"이 한 묶음으로 보여야 한다.
- 추천은 자동 실행처럼 보이면 안 된다. 사용자가 승인하는 구조여야 한다.
- Action Stack은 Decision Rail 아래 또는 카드 하단에 붙는다.

권장 문법:
- `AI로 재계획`
- `추천안 적용`
- `수정해서 적용`
- `이번에는 보류`

### 4. Draft Canvas = AI 생성

AI가 초안, 요약, 피드백, 메시지, 루브릭, 콘텐츠를 생성한 영역이다.

UI 원칙:
- 생성물은 일반 텍스트 카드와 시각적으로 달라야 한다.
- "초안 상태"와 "공개/확정 전" 상태가 명확해야 한다.
- 편집 가능성, 버전, 교수자 말투 반영 여부, 출처/근거를 같이 보여준다.

권장 문법:
- `AI 초안`
- `교수자 말투 반영`
- `편집 후 공개`
- `출처/근거 확인`

### 5. Audit Checklist = AI 검증

AI가 결과를 검토하거나 공정성, 효과, 신뢰도, 측정 계획을 확인하는 영역이다.

UI 원칙:
- 검증은 배지가 아니라 체크리스트나 audit row로 보여야 한다.
- "통과/주의/사람 검토 필요" 상태를 명확히 구분한다.
- 특히 grading, cocreation, intervention의 신뢰성과 사람 최종 결정 원칙을 강화한다.

권장 문법:
- `공정성 검토`
- `효과 검증 계획`
- `불확실 항목 교수 검토`
- `측정 기준 보기`

## 현재 Claude 프롬프트/플랜에서 특히 조심할 점

1. `AI 생성/추천/분석/초안` 4분류만으로는 부족하다. `AI 예측`, `AI 검증/측정`, `AI 판단`이 빠지면 lecture, cocreation, grading의 핵심 의미가 흐려진다.

2. `AI` 배지를 많이 붙이면 오히려 affordance가 약해진다. 배지는 보조 정보이고, 핵심은 사용자가 "이걸 누르면 무엇이 열리고, 무엇을 결정할 수 있는지"를 즉시 이해하는 것이다.

3. `expandAiPanel()`을 단순 레이아웃 토글로 취급하면 안 된다. `근거 보기`, `재계획`, `수정 지시`, `검토`는 서로 다른 target과 mode를 가져야 한다. 가능하면 `openAiContext(targetId, mode)` 형태로 분리한다.

4. SNB 토글과 AI 패널 확장은 역할이 다르다. 사용자가 제기한 관점은 "SNB는 SNB에서 접히고, AI 패널은 AI 패널에서 확장되는 것이 자연스럽다"였다. 이후 Claude 플랜에서 "AI 패널 토글 제거, SNB 단일 토글"로 바꿀 경우, 그 결정은 반드시 UX 논리로 방어되어야 한다. 단순 구현 편의로 합치면 안 된다.

5. `시작`, `검토`, `발송`, `공개` 같은 실제 실행 CTA를 레이아웃 토글로 바꾸면 안 된다. 실행 CTA와 근거/AI context 열기는 분리되어야 한다.

6. tooltip에 중요한 근거를 숨기지 않는다. 발표/검토 맥락에서는 hover 설명이 보이지 않는다. 중요한 AI 판단, 신뢰도, 검증 상태는 화면 위에 보여야 한다.

## 화면별 적용 방향

### Student Dashboard

- "오늘 학습 경로"는 Decision Rail이 적합하다.
- 각 추천 경로에는 `왜 오늘 이 순서인가`, `대안 경로`, `부담 리스크`를 열 수 있어야 한다.
- 일반 강의 카드 나열은 줄이고, AI가 판단한 오늘의 실행 경로에 집중한다.

### Student Lecture

- AI 요약/자막은 흔한 기능이라 차별점이 약하다.
- 핵심은 `이해도 예측`, `막힘 구간 감지`, `메타인지 체크`, `근거 있는 힌트`다.
- Signal Marker와 Audit/Confidence 상태를 같이 보여준다.

### Instructor Dashboard

- 운영 현황 카드보다 "AI가 왜 이 의사결정을 먼저 올렸는가"가 중심이어야 한다.
- Decision Card에는 Decision Rail + Action Stack을 붙인다.
- 알림/일정성 요소는 줄이고, 우선순위화와 근거 공개에 집중한다.

### Instructor Co-Creation

- 가장 공들일 화면이다.
- Draft Canvas가 핵심이다. AI가 만든 3안은 일반 카드가 아니라 "초안 캔버스"로 보여야 한다.
- 교수자 말투/가치 반영, 편집 후 공개, 2주 후 효과 검증이 한 루프로 읽혀야 한다.
- 효과 검증은 Audit Checklist 또는 Measurement Plan으로 분리해서 보여준다.

### Instructor Grading

- `불확실 8건`은 테이블 필터가 아니라 상단 검토 큐로 승격한다.
- AI 채점 초안은 Draft Canvas, 공정성/루브릭 확인은 Audit Checklist다.
- "AI가 채점했다"보다 "교수가 검토해야 할 것만 줄였다"가 핵심 메시지다.

## 구현 우선순위

1. AI interaction target 모델 정리: `openAiContext(targetId, mode)` 또는 동일한 의미의 구조.
2. AI affordance type을 5개 grammar로 재분류.
3. 각 grammar별 시각 문법을 CSS class로 분리.
4. 핵심 5개 화면에서만 우선 적용.
5. Design Spec에도 같은 문법을 반영해 prototype과 spec이 충돌하지 않게 정리.

## 최종 기준

좋은 결과물은 사용자가 화면을 보자마자 다음을 이해해야 한다.

- 이 항목은 AI가 감지한 것인지, 판단한 것인지, 추천한 것인지, 생성한 것인지, 검증한 것인지
- 이 항목을 누르면 실행되는지, 근거가 열리는지, 초안을 편집하는지
- AI가 어느 정도 확신하는지
- 사람이 어디서 최종 결정하는지
- AI를 거절하거나 수정할 수 있는지

이 기준을 만족하지 못하면, 스타일이 좋아져도 논리적 UX 고도화라고 보기 어렵다.
